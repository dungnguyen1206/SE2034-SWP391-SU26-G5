I. Overview
1. User Requirements
1.1 Actors


ID
	Actor
	Description
	1
	Admin
	Manages system data (users, doctors, departments, schedules, services); views reports and dashboard 
	2
	Patient
	Registers account; books and manages appointments; views medical history and payment status 
	3
	Receptionist
	Manages appointments (confirm/cancel/walk-in); checks in patients on arrival; creates invoices and processes payment 
	4
	Doctor
	Views daily schedule; views patient medical history; records examination results, diagnosis, and prescriptions; completes visit records 
	5
	Guest
	Views public information (doctors, departments, available slots); registers new account 
	6
	Hospital Manager
	Manages hospital operations including overseeing doctors, schedules, appointments, services, and medical records; generates reports; assists Admin in system management tasks. 
	7
	Mail Service
	Sends automated notifications for appointment confirmation, reminders, and status updates

b. Descriptions


ID
	Feature
	Use Case
	Use Case Description
	

	Authenticated User
	

	

	UC-1
	

	Login account
	Users can log into the system by entering email/username and password. The system authenticates the account and redirects the user to the correct role-specific dashboard.
	UC-2
	

	Forgot password
	Users can request a password reset by email. The system sends an OTP or reset link and allows the user to set a new password.
	UC-3
	

	Change password
	Logged-in users can change their current password. The system verifies the old password and updates the new password.
	

	Admin
	

	

	UC-4
	Manage Users
	List user
	Admin can view all users in the system. The system retrieves user account information and displays it in a list. Search and filter by role/status are actions on the list screen.
	UC-5
	Manage Users
	Edit user role/status
	Admin can update a user’s role or account status when necessary. The system validates the change and saves the updated role/status.
	

	Hospital Manager
	

	

	UC-6
	Dashboard
	View manager dashboard
	This use case allows the Hospital Manager to see the daily and monthly activities of the hospital on one page. The system shows key numbers like total appointments, total patients, active doctors, and this month's revenue. It also displays today's appointment list and the list of doctors on duty. This helps the manager check how the hospital is working today.
	UC-7
	Manage Staff 
	List staff
	This use case allows the Hospital Manager to see and manage all staff accounts in the system. The page shows cards with the total numbers of staff, doctors, and receptionists. Managers can search for staff using names, emails, or phone numbers, and filter them by roles (Doctor or Receptionist). The page displays a table with staff details such as names, types, statuses, and creators. From this list, the manager can also choose to view details or edit each staff profile.
	UC-8
	Manage Staff Accounts
	Edit staff 
	This use case allows the manager to update a staff member's profile. The manager can change general details like name, phone, gender, and avatar, or role-specific details like department and degree for doctors.
	UC-9 
	Manage Staff Accounts
	View staff 
	This use case allows the manager to view the detailed information of a staff member. For receptionist accounts, the manager can see personal details, while for doctor accounts, the manager can also see their appointment statistics, such as upcoming, completed, and canceled appointments.
	UC-10
	Manage Services
	List service
	This use case allows the manager to view, search, and filter all medical services in the system. The manager can filter services by department, search by keyword, and see details like service name, price, duration, and status in a table.
	UC-11
	Manage Services
	Edit service
	This use case allows the manager to update medical services information. The manager can modify fields like price, description, estimated duration, status, and upload a new service image.
	UC-12
	Manage Services
	View service
	This use case allows the manager to view the detailed information of a specific medical service. The manager can see details such as the service name, price, estimated duration, description, and status.
	UC-13
	Manage Schedules
	List schedule
	This use case allows the manager to view, filter, and manage doctor work schedules by week. The manager can filter schedules by department, doctor name, or shift, change weeks, and see shift details in a calendar table.
	UC-14
	Manage Schedules
	Add schedule
	This use case allows the manager to create a new work shift for a doctor. The manager can select the department, doctor, work date, shift type, clinic room, maximum patient capacity, and write optional notes.
	UC-15
	Manage Schedules
	Edit schedule
	This use case allows the manager to modify an existing doctor's work shift. The manager can update details like the work date, shift type, clinic room, patient capacity, status, and write notes.
	UC-16
	Manage Articles
	View list article
	This use case allows the manager to view, search, and filter all public articles and news in a list. The manager can search by keyword, filter by category or status, and view details like title, author, and status in a table.
	UC-17
	Manage Articles
	Add article
	This use case allows the manager to write and publish a new article or news. The manager can input details like title, summary, content, category, select a doctor as the author, and set the status to draft or published.
	UC-18 
	Manage Articles
	Edit article
	This use case allows the manager to edit an existing article or news. The manager can update fields like title, summary, content, category, author, and status, and save the changes to the system.
	UC-19  
	Invoice Overview
	List invoice
	This use case allows the manager to view, filter, and track all hospital invoices. The manager can filter invoices by date range, month, or year, and see invoice summaries (paid and pending totals) along with details like invoice code, patient name, amount, and payment status in a table.
	

	Receptionist
	

	

	UC-37
	Appointment Management
	View list appointment
	Allows receptionists to view the list of all appointments booked online by patients. The system provides search (by name or phone number) and data filtering (by date or status) functions, enabling receptionists to easily track and manage the volume of incoming patients.
	UC-38
	Appointment Management
	View details appointment
	This allows the receptionist to view detailed information about a specific appointment, including the patient's profile, the attending physician, the medical specialty, and the scheduled time slot. This screen also serves as the primary interface for the receptionist to handle the check-in process when the patient arrives at the clinic.
	UC-39
	Check-in
	Check-in & Print ticket
	When the patient arrives at the clinic, the receptionist confirms their check-in. The system updates the appointment status from "Confirmed" to "Waiting," adds the patient to the consultation queue, and automatically prints a queue number slip for them.
	UC-40
	Queue Management
	View list queue
	Allows the reception staff to view the list of patients who have completed check-in, including those currently waiting for or undergoing examinations today. This function enables the reception team to monitor patient flow and track the progress of medical consultations at the clinic.
	UC-41
	Invoice Management
	View list invoices
	Allows receptionists to view the list of all service invoices generated by doctors following consultations. The system supports patient searches (by name or phone number) and payment status filtering (Paid/Unpaid), enabling receptionists to easily handle cashiering tasks.
	UC-42
	Invoice Management
	View details invoice
	Allows the receptionist to view detailed information for a specific invoice. This includes patient details, a list of medical services rendered, consultation fees, the total amount due, and the current payment status of the invoice.

	UC-51

	UC-53

	UC-55

	Patient can change appointment date or time based on the doctor’s available schedule.
	UC-59

	Receptionist
	

	

	

	

	Dashboard screen
	The screen displays an overview of the day's activity statistics and a list of today's appointments. Receptionists can quickly monitor workflow progress and use the search function to rapidly look up scheduled appointments for the day.
	SCR-37
	Appointment Management
	Appointment list screen
	This screen displays the complete list of appointments. It enables receptionists to search by keyword (name or phone number) and flexibly filter the list based on date range (From/To) and status. Additionally, the screen provides a visual summary of appointment statistics categorized by status (Confirmed, Waiting, Examining, Completed, Cancelled, No Show).
	SCR-38
	Appointment Management
	Appointment detail screen
	This screen displays detailed information about a specific appointment, including the patient's profile, the attending physician, and the appointment time. From here, the receptionist can review the details and navigate to the check-in interface to process the patient's arrival and print their examination slip.
	SCR-39
	Appointment Management
	Check-in & Print ticket
	A dedicated screen displays the appointment details. The system automatically verifies the necessary conditions (the appointment must be scheduled for the current day and have a 'CONFIRMED' status). If these conditions are met, the receptionist can perform the check-in action (changing the status to 'WAITING') and print a queue ticket.
	SCR-40
	Queue Management
	Queue list screen
	The Queue Board displays the daily patient queue at the clinics. Reception staff can monitor the list of patients who have successfully checked in—whether they are currently waiting or undergoing examination in specific rooms—thereby effectively managing patient flow.
	SCR-41
	Invoice Management
	Invoice list screen
	Service invoice list management screen. It provides receptionists with tools to search for invoices by keyword (patient information) and filter the list by payment status. Data is paginated for easy management.
	SCR-42
	Invoice Management
	Invoice detail screen
	This screen displays the details of an invoice, including the list of medical services the patient has received and the total amount due. Here, the receptionist can select a payment method and process the payment to update the invoice status.

b. Table Descriptions
No
	Table
	Description
	1
	provinces
	Provinces & Cities Lookup: Stores administrative divisions in Vietnam. Contains province names and types (defaults to 'PROVINCE'). Serves as a reference lookup table for user addresses to enforce geographical data consistency.
	2
	roles
	User Roles: Defines access levels and system permissions. Stores unique system roles (ADMIN for system settings, MANAGER for hospital schedules, DOCTOR for clinical work, PATIENT for portal clients, and RECEPTIONIST for billing/check-ins) with descriptions.
	3
	departments
	Hospital Departments: Represents specialized departments (e.g., Pediatrics, Cardiology). Stores department name, description, landing page image URL, and active status (ACTIVE/INACTIVE) to determine availability on public-facing portals.
	4
	rooms
	Hospital Rooms: Represents physical consultation or diagnostic rooms. Each room belongs to a specific department (department_id FK), and contains a room name, code, status, and creation/update audit timestamps.
	5
	users
	Unified Accounts: The core table storing credentials (username, email, password hash, status, email verification flags) and common profile data (names, phone, gender, avatar). It houses role-specific fields: date_of_birth/blood_type for Patients, and department_id/license_issue_date/degree/license_number/bio/doctor_status for Doctors.
	6
	user_roles
	User-Role Mappings: A junction table resolving the many-to-many relationship between users and roles. Maps which accounts possess which privileges. Enforces cascading deletes to clean up assignments if a user or role is deleted.
	7
	user_addresses
	User Addresses: Manages mailing or contact addresses for users. Connects a user (user_id) to a province (province_id) with fields for detailed address lines and a default flag (is_default) to mark the primary billing/contact address.
	8
	medical_services
	Medical Service Catalog: Defines clinical offerings (e.g., General Examination, Blood Test, X-Ray) associated with departments. Stores service names, reference pricing, estimated duration in minutes (defaults to 30), descriptions, media URLs, and active statuses.
	9
	week_schedules
	Weekly Hospital Schedules: Controls the scheduling lifecycle on a week-by-week basis. The schedule covers Monday to Sunday and moves through statuses: DRAFT (manager planning), PUBLISHED (visible to doctors), FINALIZED (bookable by patients), or EXPIRED (due date passed without finalized release). It enforces a unique constraint on the week start date.
	10
	doctor_schedules
	Doctor Work Shifts: Links doctors to specific rooms on calendar dates under a weekly schedule sheet (week_schedule_id). Defines shift times (MORNING, AFTERNOON, FULL_DAY), manager notes, and statuses (ACTIVE or CANCELLED).
	11
	time_slots
	Appointment Time Slots: Sub-divided booking windows within a doctor's work shift (e.g., 07:00-09:00, 09:00-12:00). Tracks current occupancy (booked_capacity), max limit (max_capacity), availability status (AVAILABLE/FULL), and an optimistic locking version count to handle concurrent bookings safely.
	12
	appointments
	Patient Bookings: Records consultation bookings made by patients. Stores booking codes, patient, doctor, and slot IDs, check-in timestamps, and a status workflow: PENDING (booking pending), CONFIRMED (confirmed), IN_PROGRESS (check-in complete and paid), COMPLETED (examination done), or CANCELLED (canceled booking).
	13
	medical_records
	Patient Medical Records: Stores clinical outcomes and diagnosis documents (strict 1-1 relationship with appointments). Contains vital signs (heart rate, blood pressure, blood glucose, weight), symptoms, primary diagnosis, conclusion, prescriptions, notes, and records status (DRAFT vs. FINALIZED).
	14
	medical_service_orders
	Diagnostic Test Orders: Tracks lab tests, X-rays, or specialized procedures ordered by doctors during a consultation. Records the service price at the time of ordering, results, and order statuses: PENDING_PAYMENT (not paid), PAID (paid, ready for execution), COMPLETED (results uploaded by doctors), or CANCELLED (refunded/dismissed).
	15
	invoices
	Billing Invoices: Tracks payments made for appointments (allows 1-to-N invoices per appointment: e.g., Invoice #1 for clinical check-in fee, Invoice #2+ for diagnostic tests ordered during the visit). Tracks billing code, total amount, payment method, payment status (UNPAID, PAID, REFUNDED), and timestamp.
	16
	invoice_items
	Invoice Line Items: Lists itemized lines for services charged inside an invoice. Includes applied price, quantity, and line totals. It links to medical_service_orders via a nullable key (NULL for check-in exams, NOT NULL for diagnostic orders), protecting unique order-to-invoice relationships with a filtered index.
	17
	articles
	Articles: Stores news, blogs, and public health articles. Combines news and knowledge base contents by using a category field. Tracks title, SEO slug, content summary, authoring doctor, creator, view count, publishing status (DRAFT/PUBLISHED), and release dates.
	18
	email_logs
	Transactional Email Logs: Audits system-sent emails (booking confirmations, verification links, alerts). Records email type, recipient address, email subject, send status, exception messages, and polymorphic relation keys (related_entity_id, related_entity_type) to link logs to specific system objects.
	19
	notifications
	In-App Notifications: Stores notifications shown to users on their web or mobile dashboards. Holds the user ID, title, rich message body, notification category, read/unread flag, and polymorphic references for interactive deep-linking.

2. Limitations & Exclusions
1. Limitations 


* LI-1: The system supports only web browsers and does not provide a mobile application. 
* LI-2: The system does not support offline operation.
* LI-3: The system supports only a single hospital. 


2.Exclusions


* EX-1: Online payment is not included in this project. 
* EX-2: The system does not send SMS notifications. ( HÌNH NHƯ MÌNH CHỈ DÙNG GMAIL NHỈ)
* EX-3: The system does not support telemedicine or video consultations.
* EX-4: The system does not support direct communication between patients and doctors.

ID
	Category
	Rule Definition
	BR-C-01
	Constraints
	If the username already exists, the system shall reject the account creation. 
	BR-C-02
	Constraints
	If a password is created or updated, the system shall store it using BCrypt hashing. 
	BR-C-03
	constrains
	If an account has the ADMIN role, the system shall not allow it to be locked.

	Constraints
	If user roles are updated, the system shall replace all existing role assignments with the new ones.

	Constraints 
	If the current user does not have the ADMIN and MANAGER role, the system shall deny access to account management functions. 
	BR-C-06
	Constraints 
	If the user is not authenticated as a patient, the system shall deny the appointment booking request 
	BR-C-07
	Constraints 
	If the selected appointment date is in the past, the system shall reject the booking request. 
	BR-C-08
	Constraints 
	If the selected slot is unavailable, the system shall reject the booking request. 
	BR-C-09
	Constraints 
	If a patient already has a booking in a specific time slot, the system shall prevent the patient from booking that same slot again, unless the existing appointment status is CANCELLED or NO_SHOW.
	BR-C-10
	Constraints 
	If the booking is successful, the system shall create an appointment with the status CONFIRMED. 
	BR-C-11
	Constraints 
	If an appointment is successfully booked, the system shall increment the slot's booked capacity by 1. If the booked capacity reaches the maximum capacity, the system shall automatically set the slot status to FULL.
	BR-C-12
	Constraints 
	If an appointment does not belong to the current patient, the system shall deny access to the appointment details. 
	BR-C-13
	Constraints 
	If a patient attempts to modify appointment information, the system shall reject the request. 
	BR-C-14
	Constraints 
	If an appointment has been created, the system shall preserve its history and shall not permanently delete it. 
	BR-C-15
	Constraints 
	A doctor have only one doctor’s schedule in a day
	BR-C-16
	Constraints 
	Only authorized users (e.g., Hospital Manager) can view the full details of other employees. 
	BR-C-17
	Constraints 
	Managers can only update profile details; changing an employee's core system role classification is restricted on this screen. 
	BR-C-18
	Constraints 
	Calendar boundaries must start on Mondays and terminate on Sundays (week_start_date / week_end_date validation). 
	BR-C-19
	Constraints 
	The system shall allow only one doctor to be assigned to a specific room during the simulation.
	BR-C-20
	Constraints 
	The system shall restrict schedule modifications when the target weekly schedule status is FINALIZED (only allowing status changes to CANCELLED), and shall deny all modifications if the week schedule is EXPIRED
	BR-C-21
	Constraints 
	The phone number and doctor's license number must be unique in the system.
